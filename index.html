Visual Place Recognition

Show the system a photo and it answers a deceptively hard question: where was this taken? Not from GPS — from the pixels alone, by matching the image against a database of places it has seen before. I rebuilt it from scratch because it's the relocalisation trick that lets a robot or a phone know where it is when the satellite signal is gone.

What it is

Knowing a place from a single picture

Visual place recognition turns "where am I?" into a search problem. Every known location is stored not as a coordinate but as a compact numerical fingerprint of what it looks like. When a new query image arrives, you fingerprint it the same way and find its nearest neighbours in the database — the closest matches are the most likely places.

What makes it genuinely hard is that the same place rarely looks the same twice: lighting shifts, seasons change, viewpoints differ, people and cars come and go. A good descriptor has to ignore all of that and lock onto the structure that actually identifies the location — the building, the skyline, the layout.

The stack

From pixels to a place ID

Two ideas do the heavy lifting: a strong descriptor, and a fast search over it.

backbone

CNN feature extractor

A convolutional network turns each image into a dense feature map that captures structure, not just colour.

descriptor

Global embedding

Pool those features into one compact vector per image — the place's fingerprint, robust to clutter and light.

index

Nearest-neighbour search

Store every reference fingerprint and find the closest matches to a query fast, even over large databases.

verify

Geometric check

Re-rank the top candidates by matching local keypoints, confirming the match is geometrically consistent.

data

Reference gallery

A curated set of labelled place images — the memory the system searches against.

metric

Recall@K

The honest score: how often the true place lands in the top-K retrieved results.

Architecture

How a place is recognised

Every query runs the same retrieve-then-verify pipeline:

  1. Build the gallery

    Fingerprint every reference image once and store the vectors in a searchable index.

  2. Fingerprint the query

    Run the new photo through the same network to get its global descriptor.

  3. Retrieve

    Find the nearest descriptors in the gallery — the candidate places.

  4. Verify

    Match local keypoints between query and candidates to reject false positives.

  5. Answer

    Return the best-verified match as the recognised place.

Reflection

What rebuilding it taught me

  • Recognition is retrieval. Reframing "what place is this" as "find the nearest fingerprint" is the whole conceptual move.
  • The descriptor is everything. A representation that survives lighting, season and viewpoint changes is the difference between working and useless.
  • Global finds, local confirms. A cheap global vector narrows the field; an expensive geometric check makes the answer trustworthy.
  • Appearance change is the real enemy. The same street at dawn and dusk are nearly different worlds to a naive pixel comparison.
  • Recall@K keeps you honest. A single "it matched!" demo means nothing; measuring top-K recall across a dataset does.
